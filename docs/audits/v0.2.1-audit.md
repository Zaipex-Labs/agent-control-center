# Auditoría zaipex-acc · PR #1 (v0.2.1-multimodal)

**Fecha**: 2026-04-20 · **Rama**: `feat/v0.2.1-multimodal` · **Head**: `0e61331` · **Revisor**: Claude (Opus 4.7)

## Resumen ejecutivo

**Veredicto**: `merge condicional`. La feature multimodal está sólida a nivel funcional y bien testeada en happy paths; sin embargo, tres de las cuatro cosas que el propio `SECURITY.md` lista como "SÍ es una vulnerabilidad" están presentes en el código (cross-project messaging, cross-project blob, y un vector de RCE local vía `role`). Son preexistentes o ampliadas por el PR, no introducidas desde cero, pero el PR las toca y es el momento de arreglarlas. Recomendación: bloquear merge hasta cerrar los 3 bloqueantes; el resto (17 hallazgos) se puede abordar en v0.2.2.

**Conteo por severidad**: 1 Crítico · 3 Altos · 8 Medios · 9 Bajos · 3 Informativos.

**Top 3 bloqueantes**:

1. **[H-1] Cross-project send-message / send-to-role** — `handleSendMessage` y `handleSendToRole` no validan que `fromPeer.project_id === b.project_id === toPeer.project_id`. **Reproducido**: Alice en `projectA` envió `"PWN mensaje cross-project"` a Bob en `projectB` y Bob lo recibió. `SECURITY.md` línea 28 lo marca explícitamente como vulnerabilidad.
2. **[H-2] Descarga anónima de cualquier blob sin autorización** — `handleDownloadBlob` solo valida el regex sha256; **cualquier proceso local** (también otros usuarios si comparten el socket por alguna razón, p.ej. túnel SSH mal configurado) descarga cualquier blob si conoce/adivina el hash. No hay ACL ni project-scoping. **Reproducido**: upload + download anónimos sin registrar peer.
3. **[H-3] Inyección de comando shell vía `agent.role` en spawn tmux** — `spawnWithTmux` interpola `first.role` / `agent.role` **sin escape** en la plantilla `execSync` de `tmux new-session/new-window`. `handleAddAgent` y `handleUpdateProject` NO llaman a `validateIdentifiers`, así que una role como `$(touch /tmp/pwn)` se ejecuta cuando el user pulsa "Power up". `SECURITY.md` línea 27 marca "RCE desde un payload del dashboard o MCP tool" como vulnerabilidad.

## 1. Blobs / multimedia

### [C-1] — `handleCreateProject` escribe el config sin validar `name` (path traversal)
**Sev**: Crítico · **Evidencia**: `src/broker/handlers.ts:370-386` — `configPath = join(PROJECTS_DIR, '${b.name}.json')` sin validación.
**Impacto**: un POST a `/api/create-project` con `name: "../../../tmp/pwned"` escribe `/tmp/pwned.json` con contenido arbitrario (el mismo objeto config JSON). Más grave si el broker corre como otro user. No es RCE directo pero sí escritura arbitraria a paths bajo los permisos del broker.
**Recomendación**: llamar `validateIdentifiers(res, { name: 'name', value: b.name })` antes del `writeFileSync`. Ya existe en `src/broker/handlers.ts:153-165`.
**Reproducir**: `curl -X POST http://127.0.0.1:7899/api/create-project -H 'Content-Type: application/json' -d '{"name":"../../../tmp/pwned"}'`.

### [H-2] — `GET /api/blobs/:hash` sin autorización (confirmado)
**Sev**: Alto · **Evidencia**: `src/broker/handlers.ts:1530-1551` — solo regex `^[a-f0-9]{64}$`.
**Impacto**: cualquier proceso local descarga cualquier blob. Violación directa del apartado "Bypass del check de `project_id`" de `SECURITY.md`. Mitigación parcial: localhost-only.
**Recomendación**: opción mínima — exigir header `X-Peer-Id` válido y cruzarlo con `blob_refs` para ver si algún mensaje del proyecto del peer referencia ese hash. Opción completa — token por-sesión desde `/api/register`. La opción mínima no rompe el dashboard porque `attachmentUrl(hash)` ya sabe de qué proyecto es.
**Reproducir**:
```bash
HASH=$(printf 'x' | curl -s -X POST :7899/api/blobs/upload \
  -H 'Content-Type: application/octet-stream' -H 'X-Filename: a.bin' \
  --data-binary @- | jq -r .hash)
curl -s -D - :7899/api/blobs/$HASH   # 200 OK, bytes
```

### [M-1] — No se verifica SHA-256 al leer
**Sev**: Medio · **Evidencia**: `src/broker/blobs.ts:78-90` — `getBlob` lee el archivo sin re-hashear.
**Impacto**: bit rot silencioso; tampering por otro proceso local con acceso a `~/.zaipex-acc/blobs/` no se detecta.
**Recomendación**: en `getBlob`, al menos en modo dev/stats, verificar que `sha256(buffer) === hash`. Log de corrupción. Para producción, cachear el hash y comparar cada N lecturas.

### [M-2] — Upload no atómico (no tmp+rename)
**Sev**: Medio · **Evidencia**: `src/broker/blobs.ts:72-74` — `writeFileSync(path, buffer)` directo.
**Impacto**: si el proceso cae mid-write, queda un archivo parcial con el nombre correcto pero contenido incorrecto. La dedup futura (`!existsSync(path)`) lo considera válido y nunca lo reescribe → bytes corruptos persistentes.
**Recomendación**: escribir a `<path>.tmp-<rand>` y `renameSync` al final. El `rename` es atómico en el mismo filesystem.

### [M-3] — `_stats` gated solo por `NODE_ENV`
**Sev**: Medio · **Evidencia**: `src/broker/handlers.ts:1557`.
**Impacto**: si el broker se empaca/ejecuta sin `NODE_ENV=production` (`acc up` tras `npm install -g zaipex-acc` por defecto no setea nada), `_stats` queda expuesto revelando `total_blobs`, `total_bytes`, `orphan_count`. Leak menor pero leak.
**Recomendación**: gate doble: env `ACC_STATS_ENABLED=1` opt-in **y** requiere `X-Peer-Id` del proyecto. Quitar reliance en `NODE_ENV`.
**Reproducido**: `curl -s http://127.0.0.1:7898/api/blobs/_stats` → `{"total_blobs":2,"total_bytes":49,"orphan_count":2}`.

### [L-1] — `blob_refs` sin `UNIQUE (blob_hash, project_id, message_id)` ni FK
**Sev**: Bajo · **Evidencia**: `src/broker/database.ts:98-107`.
**Impacto**: dos uploads concurrentes del mismo hash o un retry en `handleSendMessage` crean refs duplicados. Benigno (el contador no se rompe) pero crece la tabla. Además, si un mensaje se borra sin pasar por `handleDeleteProject`, sus refs quedan huérfanos (el PR anota que `handleDeleteMessage` es futuro — `src/broker/blob-refs.ts:46-50`).
**Recomendación**: añadir `CREATE UNIQUE INDEX idx_blob_refs_triple ON blob_refs(blob_hash, project_id, IFNULL(message_id,-1))` o `INSERT OR IGNORE` en `addBlobRef`.

### [L-2] — `getBlob` hace `readdirSync(dir)` por descarga (O(N))
**Sev**: Bajo · **Evidencia**: `src/broker/blobs.ts:78-90`.
**Impacto**: con miles de blobs, cada GET escanea todo el directorio. DoS local moderado.
**Recomendación**: guardar la extensión en `blob_refs` (ya conocemos el mime en upload) o probar las extensiones del whitelist de `EXT_BY_MIME` una a una.

### [L-3] — MIME whitelist protege de XSS (sin bug, pero subrayar en docs)
**Sev**: Bajo · **Evidencia**: `src/broker/blobs.ts:123-134` y `src/shared/attachments.ts:30-45`.
**Comportamiento real**: al descargar, `Content-Type` viene de `mimeFromExt(ext)`, que es un whitelist cerrado (png, jpg, pdf, json, txt, csv, md, bin). Cualquier upload con `Content-Type: text/html` → ext=`bin` → servido como `application/octet-stream`. **Reproducido y confirmado**. El PR se benefició de este diseño existente; es robusto. Subrayarlo en `SECURITY.md` para que no se degrade en refactors futuros.

### [L-4] — `useAttachmentUpload.addFiles` descarta `rejected` silenciosamente
**Sev**: Bajo · **Evidencia**: `src/dashboard/hooks/useAttachmentUpload.ts:27-52`, y `src/dashboard/components/Compose.tsx:72-77` (drop handler no usa el return).
**Impacto**: si el usuario arrastra un archivo de 150 MB, desaparece sin feedback.
**Recomendación**: surfacearlo como `error` con el nombre del archivo y el cap, reusando el estilo `itemError` de `PendingAttachmentStrip`.

## 2. AuthZ mensajería entre agentes / proyectos

### [H-1] — Cross-project `send_message` y `send_to_role` (confirmado)
**Sev**: Alto · **Evidencia**: `src/broker/handlers.ts:961-1053` (send-message) y `1086-1213` (send-to-role). Ambos validan existencia de peers pero NO la igualdad de `project_id`.
**Impacto**: un proceso local (o un agente que aprenda el `peer_id` de otro proyecto) puede inyectar mensajes — y attachments — en proyectos ajenos. Cambia el threading, dispara `tmuxInjectWithContext` en el pane del target, y registra refs de blob bajo un proyecto que no era el origen. Directamente listado como vulnerabilidad en `SECURITY.md:28`.
**Recomendación**: al inicio de ambos handlers, tras resolver peers:
```ts
if (fromPeer.project_id !== b.project_id || toPeer.project_id !== b.project_id) {
  return error(res, 'Peers are not in the requested project', 403);
}
```
**Reproducido**:
```bash
BASE=http://127.0.0.1:7898
A=$(curl -s -X POST $BASE/api/register -d '{"project_id":"projectA","pid":1,"cwd":"/tmp/a","role":"backend","name":"Alice"}' -H 'Content-Type: application/json')
B=$(curl -s -X POST $BASE/api/register -d '{"project_id":"projectB","pid":1,"cwd":"/tmp/b","role":"frontend","name":"Bob"}' -H 'Content-Type: application/json')
AID=$(echo $A|jq -r .id); BID=$(echo $B|jq -r .id)
curl -s -X POST $BASE/api/send-message \
  -d "{\"project_id\":\"projectA\",\"from_id\":\"$AID\",\"to_id\":\"$BID\",\"text\":\"PWN cross-project\"}" -H 'Content-Type: application/json'
# {"ok":true}
curl -s -X POST $BASE/api/poll-messages -d "{\"id\":\"$BID\"}" -H 'Content-Type: application/json'
# {"messages":[{"id":1,"project_id":"projectA","from_id":"...",...,"text":"PWN cross-project",...}]}
```

### [M-4] — `get_thread_context` / `handleGetThread` sin validación de pertenencia
**Sev**: Medio · **Evidencia**: `src/server/tools.ts:245-259` y `src/broker/handlers.ts:1351-1362, 1476-1498`. La tool MCP manda solo `thread_id`; el broker resuelve el thread globalmente (`selectThreadById('', thread_id)` en `database.ts:351-360`).
**Impacto**: un agente que haya adivinado un `thread_id` de otro proyecto lee su resumen de las últimas 10 entradas de log.
**Recomendación**: la tool debe enviar `identity.project_id`, y el handler buscar `AND project_id = ?`. Rechazar si no hay match.

### [M-5] — `set_role` sin whitelist (suplantación del tech lead)
**Sev**: Medio · **Evidencia**: `src/server/tools.ts:281-297` y `src/broker/handlers.ts:890-899`.
**Impacto**: un agente `qa` ejecuta `set_role({ role: "arquitectura" })` y desde ese momento recibe todos los `send_to_role('arquitectura', …)`. El tech lead permanente deja de ser único.
**Recomendación**: rechazar `role === ARCHITECT_ROLE` en `handleSetRole`; opcionalmente, limitar `set_role` a roles declarados en el config del proyecto.

### [M-6] — `handleAddAgent` / `handleUpdateProject` no validan `role` (habilita [H-3])
**Sev**: Medio (fundamental para [H-3]) · **Evidencia**: `src/broker/handlers.ts:388-410, 412-468`.
**Recomendación**: aplicar `validateIdentifiers` a `role` y a `name` en ambos handlers.

### [L-5] — `handleSetRole` HTTP tampoco valida
**Sev**: Bajo · **Evidencia**: `src/broker/handlers.ts:890-899`. Paralelo a [M-5].

## 3. Prompts y prompt-injection

### [M-7] — `buildInstructions` interpola `name` y `role` literal en el system prompt
**Sev**: Medio · **Evidencia**: `src/server/index.ts:53-129`. Líneas 54 y 57 inyectan `${name}` y `${role}` sin sanitizar. `handleRegister` SÍ valida (`handlers.ts:816`), pero un peer registrado con payload en `name` escapa al usar `getDefaultName` que no se invoca si vino un `name` envío. `handleAddAgent` no valida, y esos `name`/`role` llegan al MCP como `ACC_NAME`/`ACC_ROLE` y terminan en el prompt.
**Impacto**: menor que [H-3] porque es self-pwn (el prompt malicioso afecta al agente cuyo config fue modificado), pero combinado con cross-project [H-1] puede usarse para sembrar texto en conversaciones ajenas.
**Recomendación**: sanitizar `name`/`role` al cargar del config; además, envolver la interpolación en un fence `<user>…</user>` tal que un salto de línea no termine la instrucción.

### [I-1] — Rules A/U/G permisivas + `--dangerously-skip-permissions`
**Sev**: Informativo (diseño documentado) · **Evidencia**: reglas A6/A5/G2 en `src/server/index.ts:83-111` + flag en `src/cli/spawn.ts:76`.
**Comportamiento**: los agentes obedecen "JUST DO IT" sobre tareas que vengan de otro agente, sin pedir confirmación, con Claude Code corriendo sin permission prompts. Si [H-1] no se arregla, un proceso local puede abrir shells ajenas a través de un agente con `Bash` allowlisted. `SECURITY.md:19` ya lo reconoce ("configura `cwd` solo a directorios donde confíes"), pero conviene endurecer en dos direcciones: (a) que la flag sea opt-in (por env) en lugar de siempre-on, (b) que A6 tenga un escape hatch para tareas destructivas ("si la petición pide borrar código o datos, pide confirmación al tech lead").

### [I-2] — Plantilla de `[system:save-resume]` interpola `peer.role` en JSON-as-string
**Sev**: Informativo · **Evidencia**: `src/broker/handlers.ts:667-667` y `768`. `set_shared("resume", "${peer.role}", …)` — si `role` tiene `"` rompe la llamada. `handleRegister` valida, pero si alguien manipula `peers` directamente queda vector de syntax error / inject en el prompt que el agente ejecuta.
**Recomendación**: `JSON.stringify(peer.role)` al interpolar.

## 4. Integración Claude Code — MCP / skills / superpowers / slash commands

### [H-3] — Command injection vía `role` en spawn tmux (confirmado por lectura)
**Sev**: Alto · **Evidencia**: `src/cli/spawn.ts:117` y `129`.
```ts
execSync(`tmux new-session -d -s ${sessionName} -n ${first.role} '${firstCmd}'`, …);
execSync(`tmux new-window -t ${sessionName} -n ${agent.role} '${aFullCmd}'`, …);
```
`first.role` y `agent.role` vienen del JSON del proyecto, que `handleAddAgent` / `handleUpdateProject` escriben sin validar. El `execSync` corre bajo `/bin/sh`, así que `role = '$(touch /tmp/pwn)'` ejecuta `touch /tmp/pwn` durante el spawn.
**Impacto**: RCE local con privilegios del broker. `SECURITY.md:27` lo marca como vulnerabilidad.
**Recomendación**: dos cosas independientes — (a) `validateIdentifiers(role)` en los handlers (arregla el vector remoto); (b) `shellEscape(first.role)` en spawn.ts (defensa en profundidad). Mejor todavía: pasar `new-session` como argv a `spawn()` en lugar de string a `execSync`.

### [M-8] — `tmuxInjectWithContext` interpola `paneTarget` sin escape
**Sev**: Medio · **Evidencia**: `src/broker/tmux.ts:63-77`. `tmux send-keys -t ${paneTarget}` donde `paneTarget = 'acc-' + projectId + ':' + targetRole`. `projectId` viene del config (validable) y `targetRole` viene de `toPeer.role` que puede haber cambiado vía [M-5] `set_role` sin whitelist.
**Recomendación**: además de arreglar `set_role`, envolver `paneTarget` con `shellEscape`, o mejor usar `execFileSync('tmux', ['send-keys', '-t', paneTarget, …])` para eliminar la plantilla shell.

### [I-3] — `--dangerously-skip-permissions` + `--dangerously-load-development-channels` siempre-on
**Sev**: Informativo · **Evidencia**: `src/cli/spawn.ts:76-77`.
**Comentario**: es el comportamiento intencional actual pero es también la mayor lista de víctimas ante [H-1]/[H-3]. La flag `skip-permissions` desactiva todas las confirmaciones de Bash/Write/Edit en los agentes, así que "pasó algo malo" → no hay señal al user antes de que ocurra.
**Recomendación**: env `ACC_UNSAFE=1` para opt-in. Para v0.3, usar `settings.json` por-agente con allowlist curada por rol (ver Anexo A).

### [B-1] — No hay `.mcp.json` ni `CLAUDE.md` por-agente salvo el arquitecto
**Sev**: Información / Bug técnico · **Evidencia**: `src/cli/spawn.ts` no escribe MCP config ni CLAUDE.md para roles no-arquitecto. Los agentes heredan del `~/.claude/` global del usuario.
**Impacto**: los agentes no-arquitecto no ven skills, comandos personalizados, ni permisos curados. La política A/U/G vive en `instructions` pasadas por flag — funciona, pero no permite al equipo publicar plantillas por-rol versionadas en el repo. Ver RFC en **Anexo A**.

## 5. SQL / persistencia / integridad

### [L-6] — `searchThreads` usa `LIKE %${query}%` sin escapar metacaracteres LIKE
**Sev**: Bajo · **Evidencia**: `src/broker/database.ts:444-452`. La query es prepared statement (parámetros ✓, no hay SQL-inject) pero `%` y `_` en el input del user hacen match amplio. Benigno en búsquedas de chat, no-bug de seguridad.

### [L-7] — `handleBrowse` + estático del dashboard: path traversal defensivo débil
**Sev**: Bajo · **Evidencia**: `src/broker/handlers.ts:185` y `src/broker/index.ts:192`. Ambos usan `.replace(/\.\./g, '')`, que **no** detiene patrones como `....//` que tras replace quedan `..//`. Mitigación real: ambos endpoints solo leen, y el dashboard estático sirve desde `DASHBOARD_DIR` (contenido confiable del build propio). Aun así, la defensa está rota. Alternativa: `resolve(base, user)` + `startsWith(base)`.

### [I-4] — Sin FKs en ningún `project_id`
Diseño deliberado (SQLite, simplicidad). La cascada la hace `deleteProjectData` en transacción (`database.ts:421-430`). OK por diseño.

# Anexo A — RFC: skills / superpowers / slash-commands en los agentes

## A.1 Estado actual

- MCP `zaipex-acc` registrado con `--scope user` (`src/cli/spawn.ts:50-59`). Disponible globalmente para cualquier sesión de Claude Code.
- Cada agente se lanza con `claude --dangerously-skip-permissions --dangerously-load-development-channels server:zaipex-acc --instruction "<A/U/G>"` (`spawn.ts:74-85`).
- Las skills / superpowers / slash-commands **del usuario** (`~/.claude/skills/**`, `~/.claude/commands/**`, settings globales) se heredan, porque Claude Code los toma de `~/.claude`.
- Solo el arquitecto recibe `CLAUDE.md` vía `ensureTechLeadDir` (`handlers.ts:319`).

**Consecuencia**: un agente `backend` del equipo ve los mismos `/plan`, `/review`, `superpowers:systematic-debugging`, etc. que ve el usuario dueño del host — siempre que éste los tenga instalados. El ACC no garantiza nada.

## A.2 Gap analysis por rol

## A.3 Propuesta concreta (scope v0.3)

1. **Crear `src/shared/claude-templates/`** en el repo, con:
   ```
   claude-templates/
     arquitectura/CLAUDE.md       (ya existe lógicamente, formalizar)
     arquitectura/settings.json
     backend/CLAUDE.md
     backend/settings.json
     frontend/CLAUDE.md
     frontend/settings.json
     qa/CLAUDE.md
     qa/settings.json
     security/CLAUDE.md
     security/settings.json
     common/skills/acc-send-message/SKILL.md  (cómo usar la tool MCP bien)
   ```
2. **Extender `spawnWithTmux` → `buildAgentCwd(agent)`** para que, antes de lanzar, escriba `<agent.cwd>/.claude/{CLAUDE.md,settings.json}` con el template del rol (merged con customización del usuario si existe). Patrón ya usado para el arquitecto.
3. **Eliminar `--dangerously-skip-permissions` por default**. Reemplazar por el `settings.json` per-rol con `permissions.allow` adecuado. Añadir flag `ACC_UNSAFE=1` para mantener el comportamiento actual durante transición.
4. **Nuevas tools MCP** (opt-in, tras discusión):
   - `list_role_skills(role)` → lista de skills/commands propuestos para un rol.
   - `notify_user(text)` → canal directo al dashboard (hoy se simula con `send_message to=user-peer-id`; formalizarlo).
5. **Cross-proyecto slash-command routing**: NO. Los slash-commands son locales al agente que los invoca; no tiene sentido enrutar `/review` entre agentes — para eso existe `send_to_role('qa', "revísame este diff")`.
