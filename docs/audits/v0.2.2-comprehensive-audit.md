## 0. Resumen ejecutivo

**Veredicto**: la base v0.2.2 cierra correctamente los 4 hallazgos críticos previos (C-1, H-1, H-2, H-3 verificados ✓). Pero la auditoría amplia destapa **dos High de seguridad nuevas que rompen el threat model** (CSRF y WS-hijacking sin Origin check; **WS-hijack es RCE local desde cualquier web**) y **~15 Medium repartidos** entre cross-project leaks de endpoints no parcheados (shared/get, get-history, threads), arquitectura del broker (god file 1676 LOC), perf (sin prepared-statement cache, sync I/O en hot path), y UX (mobile roto, /history vacío, Google Fonts externos). El system prompt MCP gasta ~1,920 tokens × cada turno × cada agente, ~50% reducible sin perder comportamiento.

Convención de severidad usada en TODO el documento (válida también para UX y perf, no solo seguridad):
- **Crítico**: rompe el threat model **y** es trivial de explotar.
- **Alto**: rompe threat model, bloquea un flujo central del producto, o degrada severamente seguridad / perf / UX.
- **Medio**: bug observable, deuda significativa, o degradación moderada.
- **Bajo**: pulido, edge case, deuda menor.
- **Info**: observación de arquitectura o reducción de tokens.

**Top 5 que deberían ir en el próximo PR (`v0.2.3`)**:
1. **`S-NEW-2`** — WebSocket sin verifyClient + sin Origin check → cualquier web visitada por el usuario puede pipelinear bytes al stdin del agente y leer su stdout. RCE local desde el navegador.
2. **`S-NEW-1`** — Endpoints HTTP del broker sin Origin/Host check → CSRF + DNS-rebinding contra `127.0.0.1:7899`.
3. **`S-NEW-3`** — `/api/shared/get|set`, `/api/get-history`, `/api/threads/*`, `/api/save-resume`, `/api/list-modified-files` no validan `peer.project_id === body.project_id`. El fix de H-1 sólo cubrió messaging.
4. **`S-NEW-4`** / `M-5` / `L-5` — `handleSetRole` acepta cualquier string → impersonation de `arquitectura`, intercepta `send_to_role`.
5. **`P-12`** — Sin `SIGTERM`/`SIGINT`/`uncaughtException` handlers en el broker → crash deja agentes huérfanos, tmux sin reapear, WAL sin checkpoint.

## 3. Seguridad — hallazgos NUEVOS (no estaban en v0.2.1)

> Threat model en `SECURITY.md`: localhost-only, no auth por diseño. Lo que SÍ es vulnerabilidad: cross-project leak, RCE desde dashboard/MCP, broker en 0.0.0.0, escape del cwd de un agente.

### Alto

- [ ] **S-NEW-1 · CSRF + DNS-rebinding contra el broker HTTP** · *Alto*
  - **Qué**: `src/broker/handlers.ts:97-120` (`parseBody`) acepta cualquier `Content-Type`; `src/broker/index.ts:139-224` no chequea `Origin` ni `Host`, no responde `OPTIONS`.
  - **Por qué**: cualquier web que el usuario abra puede emitir POST cross-origin a `127.0.0.1:7899` como "simple request" (Content-Type text/plain), sin preflight. Combinado con un `peer_id` conocido o sondeable, dispara `/api/send-message`, `/api/shared/set`, `/api/project/down`, `/api/project/delete`. Con DNS rebinding lee respuestas también.
  - **Para qué (fix)**: rechazar Content-Type ≠ `application/json` en `parseBody`; rechazar `Host` ≠ `127.0.0.1:port|localhost:port`; rechazar `Origin` ≠ localhost en rutas state-changing.
  - **Repro**: `<script>fetch('http://127.0.0.1:7899/api/project/delete', {method:'POST', headers:{'Content-Type':'text/plain'}, body:JSON.stringify({project_id:'x'})})</script>` desde una página externa.

- [ ] **S-NEW-2 · WebSocket-hijacking → RCE vía stdin del terminal del agente** · *Alto*
  - **Qué**: `src/broker/websocket.ts:26-41` y `src/broker/terminal.ts:466-516` crean `WebSocketServer` sin `verifyClient` y el `upgrade` handler no inspecciona `Origin`. WebSockets no están bajo Same-Origin Policy a nivel de red (no hay preflight CORS).
  - **Por qué**: una página servida desde otro origin hace `new WebSocket('ws://127.0.0.1:7899/ws/terminal/<role>?project=<id>')`, lee stdout del agente (código, secretos del cwd, output de Claude) y escribe stdin → con `--dangerously-skip-permissions` activo (siempre on en spawn web, ver `terminal.ts:166-167`) es **RCE local instantáneo** dentro del cwd del agente. SECURITY.md ya marca esto como vulnerabilidad.
  - **PoC**: `docs/audits/qa-out/poc-s-new-2.html`. Servirlo desde otro puerto (`cd docs/audits/qa-out && python3 -m http.server 8080`) y abrir `http://127.0.0.1:8080/poc-s-new-2.html` en el mismo Chrome donde corre el broker. El payload por defecto es inocuo (`echo hijacked-by-poc\n`).
  - **Verificación del handshake (sin agente activo)**:
    ```
    $ node -e "const W=require('ws'); const w=new W('ws://127.0.0.1:7899/ws/terminal/backend?project=zaipex-sass',{headers:{Origin:'http://attacker.local:8080'}}); w.on('open',()=>console.log('HANDSHAKE OK')); w.on('close',(c,r)=>console.log('close',c,r.toString()));"
    HANDSHAKE OK   ← acepta Origin externo
    close 1011 Agent not running   ← solo cierra porque no hay proceso, no por la auth
    ```
    Con un agente vivo la conexión queda abierta y `ws.send(stdin)` se entrega.
  - **Vectores REALES** (cuáles aplican y cuáles no):
    - ✅ **Página HTTP plain en `localhost:*` o `127.0.0.1:*`** (otros dev servers, herramientas locales, docs servidos en `python3 -m http.server`). Más común de lo que parece — cualquier dev tiene 3-4 puertos vivos.
    - ✅ **Página HTTP plain en LAN/Internet abierta directamente con `file://` o `http://attacker.test`** (sin TLS). El usuario tiene que estar en esa página, pero un malicioso link en Discord/Slack/email basta.
    - ✅ **Extensión de navegador maliciosa o comprometida** (Chrome extension, Firefox add-on). No tiene restricciones de origin.
    - ✅ **App Electron de terceros** corriendo en la misma máquina. Mismo navegador, mismo proceso, mismo localhost.
    - ✅ **Iframe HTTP dentro de página HTTP**.
    - ❌ **Página HTTPS** (atacker.com con TLS) → `ws://` está bloqueado por *mixed content* en navegadores modernos (Chrome, Firefox, Safari). Tendrían que conseguir que sea `wss://`, lo cual exige TLS local, lo cual no aplica al broker.
    - ❌ **Iframe HTTPS dentro de página HTTPS** → mismo bloqueo mixed-content.
    - ⚠️ **`http://attacker.com` cargado desde un router/proxy local** → posible vía DNS rebinding (resuelve `attacker.com` a `127.0.0.1` después del primer request) — relacionado con `S-NEW-1`.

    El vector HTTP-plain sigue siendo trivialmente viable porque (a) es lo que sirve cualquier dev server local, (b) la promesa "local-only" de la herramienta hace pensar que `ws://localhost:*` es seguro, y (c) un dev típico tiene 5+ puertos HTTP plain abiertos en cualquier momento. **La severidad se mantiene en Alto.**
  - **Para qué (fix)**: `verifyClient` en ambos `WebSocketServer` que valide `req.headers.origin ∈ {undefined, /^https?:\/\/127\.0\.0\.1(:\d+)?$/, /^https?:\/\/localhost(:\d+)?$/}`. `undefined` permite clientes no-browser (curl, scripts node) que es como funcionan los tests. Para `/ws/terminal/*`: además exigir `X-Peer-Id` registrado y proyecto coincidente — capa extra contra navegador local mal portado.

### Medio

- [ ] **S-NEW-3 · Cross-project leak en endpoints NO cubiertos por el fix H-1** · *Medio*
  - **Qué**: `handleSharedSet/Get/List/Delete` (`handlers.ts:1330-1370`), `handleGetHistory` (`handlers.ts:1313-1326`), `handleListModifiedFiles` (`handlers.ts:611-668`), `handleSaveResume` (`handlers.ts:680-734`), `handleGetThread` (`handlers.ts:1417-1428`), `handleDeleteThread` (`handlers.ts:1456-1467`), `handleThreadSummary` (`handlers.ts:1542-1564`) aceptan `project_id` del body sin validar que el `peer_id` (cuando lo hay) pertenezca al mismo proyecto.
  - **Por qué**: el fix H-1 sólo bloqueó messaging. Un peer en proyecto A puede pedir `/api/shared/get?project_id=B&namespace=contracts&key=db_password` o el log entero de B con `/api/get-history`. SECURITY.md prohíbe explícitamente "Bypass del check de project_id".
  - **Para qué**: aplicar el mismo patrón de `PROJECT_MISMATCH` (403) en cada handler. Helper compartido `assertProjectMembership(peer_id, project_id)`.

- [ ] **S-NEW-4 · `handleSetRole` acepta cualquier string → architect-impersonation** · *Medio*
  - **Qué**: `handlers.ts:928-937` no valida formato ni reserva `arquitectura`.
  - **Por qué**: un peer `qa` puede `set_role: 'arquitectura'` y empezar a recibir todos los `send_to_role('arquitectura', …)` (incluso de otros agentes). Mata el routing de coordinación.
  - **Para qué**: `assertSafeIdentifier('role', b.role)` + lista cerrada (`config.agents.map(a => a.role)`) o blacklist `ARCHITECT_ROLE`.
  - Ver también **M-5** y **L-5** (mismo síntoma desde MCP y CLI; quedaron abiertos de v0.2.1).

### Bajo

- [ ] **S-NEW-5 · Path traversal `....//` en `handleBrowse` y static serve** · *Bajo*
  - **Qué**: `handlers.ts:193` y `index.ts:192` ambos hacen `safePath.replace(/\.\./g, '')` — `....//` → `..//` sobrevive y `join` sale del directorio.
  - **Por qué**: explotabilidad limitada (sólo lee, sólo `.html/.js/.css/.svg/.png/.woff/.woff2/.ico` por whitelist MIME, falback a SPA), pero principio de defensa rota. Carry-over de L-7.
  - **Para qué**: `realpath` + `startsWith(BASE + sep)`.

- [ ] **S-NEW-6 · Sin cap en `attachments[]` por mensaje → DoS local + bloat** · *Bajo*
  - **Qué**: `handlers.ts:1036-1048, 1172-1184` iteran `incoming` sin tope. 1MB body ≈ ~5,000 descriptores; cada uno hace `getBlob()` (que es `readdirSync`, ver L-2) + `INSERT`.
  - **Para qué**: `if (incoming.length > 32) return error(res, 'Too many attachments', 400)`.

- [ ] **S-NEW-7 · `peer.role` se interpola en plantilla JSON-as-string del prompt resume** · *Bajo (prompt-injection contingente)*
  - **Qué**: `handlers.ts:705, 806` — `set_shared("resume", "${peer.role}", JSON.stringify({...}))`.
  - **Por qué**: combinado con `S-NEW-4`/`M-5`, un role como `x")\n;rm -rf /tmp/x;//` rompe el JSON template. Reproducible vía `/api/set-role`.
  - **Para qué**: `JSON.stringify(peer.role)` para envolver, o templating con escape consciente de contexto.

- [ ] **S-NEW-8 · `handleGetThread` admite `project_id=''` → cualquier thread brute-forceable** · *Bajo*
  - **Qué**: `handlers.ts:1417-1428` llama `selectThreadById('', b.thread_id)`; `database.ts:351-360` no filtra cuando project_id es vacío. `generateId()` da 8 hex chars (~32 bits, brute-forceable local).
  - **Para qué**: requerir `project_id` no-vacío y validarlo. Igual en `handleUpdateThread` (`handlers.ts:1430-1450`).

- [ ] **S-NEW-9 · `handleDeleteThread` no libera `blob_refs`** · *Bajo*
  - **Qué**: `handlers.ts:1456-1467` no llama nada equivalente a `releaseBlobRefsForMessage`.
  - **Por qué**: leak de disco (no de seguridad), agrava L-2/M-2.

- [ ] **S-NEW-10 · Terminal WS sin rate-limit ni backpressure** · *Bajo*
  - **Qué**: `src/broker/terminal.ts:47, 207-216`. `MAX_BUFFER` está bien pero no hay tope en flujo `proc.stdout → ws.send` ni rate-limit en stdin.
  - **Para qué**: drop si `ws.bufferedAmount > 1MB`; token bucket en stdin.

## 5. Backend — performance, errores, lifecycle (20 hallazgos)

### Alto

- [ ] **P-1 · Sin caché de prepared statements** · *Alto* (relativo) / *Bajo* (absoluto) — leer benchmark
  - **Qué**: `src/broker/database.ts` re-prepara cada statement por llamada (~53 sitios). `handlePollMessages` ejecuta `selectUndelivered` + `markDelivered`×2 cada ~1s × N agentes.
  - **Benchmark** (`scripts/bench/p1-prepared-stmt.mjs`, 10k mensajes undelivered, 1000 iteraciones, in-memory DB):
    ```
    --- selectUndelivered (read, returns ~2500 rows per peer) ---
    label                    median       p95         p99         mean
    reprep selectUndeliv     94.50 µs     105.71 µs   156.25 µs   97.38 µs
    cached selectUndeliv     91.25 µs     102.63 µs   116.96 µs   93.65 µs
    Δ (re-prepare vs cached): median 3.6%  p95 3.0%  p99 33.6%

    --- selectUndelivered LIMIT 10 (realistic poll page) ---
    label                    median       p95         p99         mean
    reprep selectUnd-LIMIT10 13.71 µs     16.63 µs    23.04 µs    14.59 µs
    cached selectUnd-LIMIT10  9.21 µs     10.46 µs    13.17 µs    10.12 µs
    Δ (re-prepare vs cached): median 48.9%  p95 59.0%  p99 75.0%

    --- markDelivered IN(?…?) 10 ids (write) ---
    label                    median       p95         p99         mean
    reprep markDelivered      8.38 µs     11.04 µs    13.92 µs    8.85 µs
    cached markDelivered      5.67 µs      6.50 µs     8.13 µs    5.82 µs
    Δ (re-prepare vs cached): median 47.8%  p95 69.9%  p99 71.3%

    --- prepare() cost in isolation ---
    db.prepare() only: median 3.83 µs   p95 5.25 µs   p99 6.96 µs
    ```
  - **Lectura del benchmark**:
    - El delta **relativo** en el hot path realista (LIMIT 10 poll y markDelivered) es **48-49% median, 59-70% p95, 71-75% p99** → por la regla del usuario (`>20%` ⇒ Alto), severidad relativa **Alto**.
    - El **costo absoluto** es microsegundos: ~5 µs ahorrados por call. Con 4 agentes × 1 Hz × 4 statements ≈ 80 µs/s ahorrados ≈ 0.008% de un core. Es ruido en cualquier máquina moderna.
    - Conclusión: **mantener Alto por la regla**, pero documentar que el ROI absoluto es modesto. Vale la pena por ergonomía (intent claro, easier to spot accidental N+1) y por consistencia con cómo better-sqlite3 quiere que se use, no por urgencia operacional. Si hay que priorizar, este se puede diferir a v0.2.4 sin riesgo.
  - **Nota técnica**: better-sqlite3 NO cachea preparaciones idénticas internamente (lo verificamos con el bench de `db.prepare()` aislado: 3.83 µs median consistent — si hubiera caché sería ~0). El delta medido es real, no observador-de-medición.
  - **Para qué (fix)**: construir `STMT = { selectUndelivered: db.prepare(...), … }` en `initDatabase` y exportarlo. Mecánico pero ~50 ediciones (esfuerzo M).

- [ ] **P-2 · Sync FS en hot path** · *Alto*
  - **Qué**: `handlers.ts:197, 231-249, 278-295, 422-437, 451-505, 513-541, 615-668, 1476-1528` usan `readdirSync`/`readFileSync`/`writeFileSync`. `handleListProjects` además dispara `tmux has-session` (fork) por proyecto.
  - **Para qué**: migrar a `fs/promises` y `execFile` async. En `handleListProjects` batchear `tmux list-sessions` una vez.

- [ ] **P-3 · `execSync('git status …', {timeout:2500})` bloquea el event loop hasta 2.5s** · *Alto*
  - **Qué**: `src/broker/files.ts:24-29` desde `handleListModifiedFiles` (`handlers.ts:633`). Multiplicado por agents.length, peor caso ~12.5s parado.
  - **Para qué**: `execFile` async + `Promise.all`. Ver QW-6.

- [ ] **P-4 · WS broadcast sin backpressure ni heartbeat** · *Alto*
  - **Qué**: `src/broker/websocket.ts:43-52` `client.ws.send(payload)` sin chequear `bufferedAmount`. Sin ping/pong → half-open TCPs no se reapean (clients sólo encoge en `'close'`/`'error'`).
  - **Para qué**: `setInterval(() => clients.forEach(c => c.ws.ping()), 30000)` con `terminate()` por pong perdido; drop si `bufferedAmount > 1MB`.

- [ ] **P-5 · Terminal WS pipea PTY sin backpressure** · *Alto*
  - **Qué**: `src/broker/terminal.ts:489-502`. `proc.stdout?.on('data', d => ws.send(d))` sin pause/resume.
  - **Para qué**: `proc.stdout.pause()` cuando `ws.bufferedAmount > 1MB`, `resume()` en `'drain'`.

### §7-bis · Análisis profundo de M-1 (compresión system prompt)

> Esta sección reemplaza la propuesta inicial "1,920 → 880 tokens" con evidencia de `git blame` y dos targets concretos. Revisada antes de editar `server/index.ts`.

**Las 5 reglas presuntamente duplicadas, con su texto literal**:

```
A4 (server/index.ts:81)
  When you receive a message from another AGENT, always respond to THAT agent —
  never directly to the user. The agent who is talking to the user is the one who
  summarizes. When you receive a message from another agent and need to respond,
  respond ONLY to that agent. Do NOT also send a message to the user saying "me
  preguntaron X". The coordination thread in the dashboard already shows the
  user what happened.

A5 (server/index.ts:83)
  If another agent messages you, it is because the user asked for it. Do the
  work. Never refuse with "that is not my area" or "ask another agent" — the
  routing already happened. Open files, run tools, check git, whatever it
  takes, and reply with the actual answer. If you genuinely cannot answer
  (e.g. the info lives in a file you do not have), say so in one line and stop
  — do not punt the question back.

A6 (server/index.ts:85)
  When an agent sends you a TASK (modify files, run commands, write code, change
  state, create a PR, etc.), JUST DO IT. Do not ask the user "should I do
  this?", do not wait for human approval, do not narrate the task to the user
  first. Agent-to-agent task requests are pre-authorized: the user started this
  chain when they talked to the other agent, and that agent decided you were
  the right one to execute. Do the work, and reply to the REQUESTING agent (not
  the user) with the result or the error. The only reason to stop is if the
  task is physically impossible (missing file, broken tool) — in that case
  reply to the agent with the blocker in one line.

G2 (server/index.ts:111)
  When you receive a message from another agent, respond IMMEDIATELY to that
  agent. NEVER surface the agent's request to the user as "should I do this?".
  NEVER wait for user approval before acting on an agent-to-agent task. The
  user has already consented by virtue of having started the conversation with
  the other agent.

G6 (server/index.ts:121)
  When you coordinate with another agent (send_message + wait for reply), the
  ENTIRE coordination is silent from the user's perspective. The user only
  sees: (a) the coordination thread (auto-generated by the dashboard), and
  (b) your final consolidated response. Nothing in between.
```

**Síntesis del análisis**:

- A4 + G6 son redundantes garantizado (mismo commit, dicen lo mismo: "agent-to-agent es silencioso desde la perspectiva del user").
- A4 + G2 (versión expandida) tienen overlap parcial: A4 dice "responde a ese agente, no al user"; G2 dice "responde inmediatamente, no pidas permiso". Comparten el "no al user" pero los matices "inmediatamente" y "no pidas permiso" son distintos.
- A5 + A6 + G2 son un cluster de refuerzo del mismo problema (rechazo / pedir permiso). Intencional pero sobre-articulado: una regla sola con 2 ejemplos breves probablemente basta.
- "Your tools" (lines 65-71) es 100% recuperable sin riesgo.
- G9 (`[system:save-resume]`, ~175 tokens) sólo dispara una vez por sesión pero está siempre en contexto. Mover a inyección dinámica (broker prepende cuando dispara) recupera tokens sin perder funcionalidad. Riesgo bajo.

**Recomendación final**:
- **Hacer el conservador (~27% recorte) sin más validación**. Riesgo bajo, ahorro ~520 tokens × cada turno × cada agente. Vale la pena solo.
- **NO hacer el agresivo sin eval-harness primero**. Crear `scripts/eval/agent-prompt-eval.mjs` con 5-10 escenarios reproducibles (agent-to-agent task, agent refusal trigger, user filler trigger, save-resume trigger, etc.), correr el prompt actual y el agresivo, comparar comportamiento. Si pasa eval, escalar.
- Documentar la decisión en `decisions.md` (cuál target tomamos, por qué, qué eval falta para el siguiente).

  - **Para qué**: media query <768px que oculte la ilustración del card, stackee header del project, y reduzca paddings. O eliminar mobile del scope y poner un mensaje "abre desde desktop".

- [ ] **UX-3 · Carga 4 recursos externos de Google Fonts en cada página** · *Medio (privacidad)*
  - **Qué**: `dist/dashboard/index.html` hace `<link rel="preconnect" href="https://fonts.googleapis.com">` + carga DM Sans, DM Serif Display, JetBrains Mono desde `fonts.gstatic.com`.
  - **Por qué**: la tool dice ser local-only (sin servicios externos, sin cuentas), pero al abrir el dashboard cada usuario filtra su IP a Google + tiempos de cada apertura. Rompe la promesa.
  - **Para qué**: hostear las 3 fuentes localmente (`@font-face` con woff2 en `dist/dashboard/assets/fonts/`), o caer a `font-family: system-ui`. Bonus: la página carga ~4 RTTs menos. Ver QW-7.

- [ ] **UX-4 · `GET /api/blobs/<bad-hash>` devuelve 200 con index.html** · *Bajo*
  - **Qué**: `src/broker/index.ts:170-173` requiere regex `[a-f0-9]{64}`; cuando no matchea, el control cae al SPA fallback (`:209-218`) y devuelve la app con 200.
  - **Por qué**: clientes/monitoring no tienen señal clara de error (ven HTML 200 cuando esperan blob 4xx).
  - **Para qué**: si la URL empieza con `/api/`, jamás devolver SPA fallback; devolver `{ok:false,error:'Not found',code:'NOT_FOUND'}` con 404.

- [ ] **UX-5 · Estado vacío del proyecto repite el mismo CTA tres veces** · *Bajo*
  - **Qué**: cuando un proyecto no tiene agentes activos, el panel central muestra "No hay agentes activos · Encender equipo", el panel izquierdo muestra "No hay agentes activos en este proyecto", y el header derecho dice "0 agentes activos". Tres mensajes idénticos.
  - **Para qué**: dejar uno (el central) y simplificar laterales a estado neutro o badge.

- [ ] **UX-6 · Header de ProjectPage no comunica estado de los botones** · *Bajo*
  - **Qué**: `Guardar` aparece en gris claro (parece deshabilitado) sin tooltip. `Apagar equipo` está habilitado incluso con 0 agentes. `Panel` no comunica qué hace.
  - **Para qué**: tooltips con razón si está disabled, esconder `Apagar equipo` si no hay nada que apagar, label del Panel.
